سیستم جامع فروشگاه آنلاین و مدیریت کیف پول دیجیتال
# Shop Project | Secure E-Commerce & Digital Wallet Platform

یک سیستم فروشگاهی سازمانی با تمرکز بر **تراکنش‌های امن، کیف پول دیجیتال، دفترکل مالی، قوانین پویا، و معماری مقیاس‌پذیر**.

این پروژه فقط یک فروشگاه آنلاین نیست؛ بلکه یک پلتفرم مالی-تجاری است که برای سناریوهای حساس مانند پرداخت، انتقال وجه، کنترل قوانین کسب‌وکار، و جلوگیری از تراکنش تکراری طراحی شده است.

## قابلیت‌های اصلی

### 1) فروشگاه آنلاین
- مدیریت کالا و دسته‌بندی‌ها
- جستجوی سریع و فیلتر محصولات
- سبد خرید و ثبت سفارش
- پرداخت با کیف پول یا روش‌های پرداخت دیگر

### 2) کیف پول دیجیتال
- شارژ کیف پول کاربر
- پرداخت از موجودی کیف پول
- انتقال وجه بین کاربران
- ثبت کامل تاریخچه تراکنش‌ها
- کنترل موجودی در لحظه

### 3) Ledger / دفترکل مالی
در این پروژه، هر تراکنش مالی فقط یک تغییر موجودی ساده نیست؛ بلکه در **دفترکل (Ledger)** ثبت می‌شود.

Ledger نقش منبع اصلی حقیقت مالی را دارد و برای هر عملیات موارد زیر را نگه می‌دارد:
- نوع تراکنش
- مبلغ
- مبدا و مقصد
- وضعیت نهایی
- شناسه یکتا
- زمان ثبت
- کاربر یا سرویس ایجادکننده

این ساختار باعث می‌شود:
- ردگیری مالی دقیق باشد
- امکان Audit وجود داشته باشد
- مغایرت مالی سریع‌تر شناسایی شود
- تراکنش‌های مشکوک قابل بررسی باشند

### 4) دفتر داخلی / Internal Book
علاوه بر Ledger عمومی، یک **دفتر داخلی** برای ثبت رویدادهای سیستمی و تسویه‌های درون‌سامانه‌ای نگهداری می‌شود.

کاربردها:
- ثبت انتقال بین کیف پول‌ها
- نگهداری رویدادهای مالی داخلی
- تسویه بین سرویس‌ها
- ثبت تغییرات موجودی قبل و بعد از عملیات

این لایه کمک می‌کند منطق مالی از منطق نمایشی جدا شود و سیستم قابل ممیزی و قابل اعتماد باقی بماند.

### 5) جلوگیری از تراکنش تکراری
برای جلوگیری از دوبار ثبت شدن تراکنش‌ها، از چند مکانیزم استفاده شده است:
- **Idempotency Key**
- **Optimistic Locking**
- در صورت نیاز **Pessimistic Locking**

#### Optimistic Locking
برای حالتی مناسب است که تداخل کم باشد. اگر دو درخواست همزمان روی یک رکورد کار کنند، نسخه‌ی قدیمی رد می‌شود.

#### Pessimistic Locking
برای عملیات حساس مثل برداشت، انتقال یا تسویه، می‌توان رکورد را قفل کرد تا همزمانی باعث خطای مالی نشود.

### 6) Rule Engine / سیستم قوانین
ادمین می‌تواند قوانین پویا برای کاربران و تراکنش‌ها تعریف کند، بدون اینکه نیاز به تغییر کد باشد.

نمونه قوانین:
- محدودیت مبلغ کارت‌به‌کارت
- محدودیت پرداخت بر اساس سن یا کد ملی
- محدودسازی خرید برای کاربران خاص
- محدودیت روزانه/ماهانه تراکنش
- فعال/غیرفعال‌سازی قابلیت‌های مالی

قوانین می‌توانند بر اساس:
- مشخصات کاربر
- سابقه تراکنش
- وضعیت حساب
- نقش کاربر
- ستون‌های اطلاعاتی سفارشی

اعمال شوند.

### 7) معماری رویدادمحور
- Kafka برای انتشار رویدادهای مهم
- مناسب برای پردازش غیرهمزمان
- مناسب برای Audit، Notification، Recommendation و Workflow

### 8) پیشنهاد کالا و تخفیف
- پیشنهاد کالا بر اساس دسته‌بندی، خریدهای قبلی و رفتار کاربر
- پیشنهاد تخفیف بر اساس سابقه خرید یا تصمیم ادمین
- تخفیف‌های زمان‌دار و قابل انقضا
